# M3 W4A16 AWQ Quantization

## Goal

Compress all Linear layer weights from bf16 to int4, reducing weight HBM read volume by 4x to accelerate decode.

---

## Data Format and Packing Convention

### int4 packing rule (nibble little-endian)

```
packed[i] = w[8i+0] | w[8i+1]<<4 | w[8i+2]<<8 | ... | w[8i+7]<<28
```

nibble 0 = lowest 4 bits = lowest index. Pack and unpack must use the same convention; mismatches will corrupt values.

### Tensor layout

| Tensor | Shape | Dtype | Notes |
|--------|-------|-------|-------|
| `qweight` | `[K//8, N]` | int32 | Packed along K dimension; nibble i of `qweight[j,n]` = weight[8j+i, n] |
| `scales`  | `[G, N]`    | bf16  | Per-group scale, G = K // group_size |
| `qzeros`  | `[G, N//8]` | int32 | Packed along N dimension; same pack_int4 convention |

---

## Dequant Formula

```
w_fp = (w_int.to(fp32) - zero.to(fp32)) * scale.to(fp32)
```

Asymmetric quantization; zero is an int4 value (0~15). The entire operation is done in fp32 to avoid precision loss from bf16 subtraction.

---

## Triton Kernel Design (w4a16_gemm)

### Grid

```
Grid = (cdiv(M, BLOCK_M), cdiv(N, BLOCK_N))
```

Each program is responsible for `out[pid_m*BM:(pid_m+1)*BM, pid_n*BN:(pid_n+1)*BN]`.

### Inner loop structure

```python
for g in range(K // group_size):          # outer: group loop (K//128 iterations)
    load scale [BLOCK_N]
    load + decode qzero [BLOCK_N]
    for j in static_range(group_size // 8):  # inner: constexpr-unrolled 16 times
        load qw [BLOCK_N] int32
        unpack 8 nibbles → w_fp0..w_fp7 [BLOCK_N] fp32
        load x0..x7 [BLOCK_M] bf16→fp32
        acc += xi[:, None] * w_fpi[None, :]  # outer product accumulation
```

### Key design decisions

1. **K=8 outer product instead of tl.dot**: SM 8.9 Tensor Cores require K>=16; with K=8, `tl.dot` falls back to CUDA Cores. Using 8 outer products is semantically equivalent and is the correct choice because decode is memory-bound and does not depend on Tensor Core throughput.

2. **Literal constants for shifts**: `(qw >> 0) & 0xF`, `(qw >> 4) & 0xF`, ... ensures Triton unrolls the loop at compile time (shift amounts are constexpr).

3. **Full fp32 dequant**: Convert to fp32 before subtraction and multiplication. If scale is bf16 and zero is subtracted directly in bf16, 1-2 bits of precision are lost at the rounding boundary.

4. **Autotune key includes group_size**: `key = ['M', 'N', 'K', 'group_size']`; different group sizes may have different optimal tile configurations.

## AWQ Quantization Theory (simplified, used for kernel testing)

This implementation uses per-group min-max quantization (without activation-aware scaling) as the data source for kernel correctness verification.

```python
scale = (w_max - w_min) / 15.0          # [G, N]
zero  = (-w_min / scale).round()         # [G, N] int4, 0~15
w_int = round(w / scale + zero)          # [K, N] int4
```

Full AWQ (Lin et al., 2023) adds activation-aware weight scaling on top of this:
1. Use calibration data to compute the activation magnitude `s_x` per column of x
2. Pre-scale each weight column by `s_x^α` (α ≈ 0.5)
3. Compensate x by dividing by `s_x^α` (can be folded into the preceding RMS Norm)
4. Goal: reduce quantization error for columns with large activation magnitudes, which have the greatest impact on output accuracy

The interface skeleton is preserved (`_per_group_minmax_quantize` in `awq.py`); the full calibration pipeline is left as an extension point.

## Measured Results (RTX 4090, K=1024, N=4096)

### Correctness (pytest)

| Test case | max abs error | Result |
|-----------|---------------|--------|
| M=1, K=128, N=8 (smallest shape) | 0.000000 | PASS |
| M=1, K=1024, N=4096 | ≤ 0.004 | PASS |
| M=4, K=1024, N=4096 | ≤ 0.250 | PASS |
| M=16, K=1024, N=4096 | ≤ 0.500 | PASS |

Error source: the fp32 kernel vs fp64 reference differs by at most 1 ULP at bf16 rounding boundaries. When magnitude is ~130, 1 ULP = 1.0; after K=1024 accumulations the maximum discrepancy is ~0.5 ULP. A real dequant bug (e.g., incorrect nibble ordering) produces O(4+) error, which is unmistakably different.

### Performance (decode scenario)

| M | BF16 cuBLAS | W4A16 Triton | speedup |
|---|-------------|--------------|---------|
| 1 | 0.008 ms | 0.051 ms | 0.16x |
| 4 | 0.009 ms | 0.058 ms | 0.15x |
| 8 | 0.009 ms | 0.057 ms | 0.16x |
| 16 | 0.009 ms | 0.058 ms | 0.16x |

**The current implementation does not meet the >= 2x performance target.**

Bottleneck analysis:
- BF16 cuBLAS reads 8 MB of weights in ~8 μs, near the HBM bandwidth limit (4090 ~1 TB/s)
- Triton kernel should theoretically read 2 MB (int4 → 4x compression → 2 μs); measured latency is 51 μs at M=1
- cuBLAS is deeply optimized (CUTLASS backend, persistent kernels, L2 prefetch); Triton cannot match it
- The gap is primarily from Triton launch overhead (~5 μs per launch) and cuBLAS's deep cache reuse

Optimizations already applied (M2→M3 iteration):
1. **GEMM kernel**: X changed to `[BLOCK_M, 16]` 2D coalesced load + `tl.dot(K=16)`, replacing 8 independent scalar loads
2. **GEMV kernel (M=1 dedicated)**: A separate 1D kernel avoids the waste of BLOCK_M=16 for single-token decode; M=1 latency reduced from 175 μs → 51 μs (3.4x improvement)
3. **K=16 combination**: Merge two rows of qweight (8 nibbles each) using `tl.where` to satisfy Triton's `tl.dot` K>=16 requirement

**Production-level further optimization directions** (as used by ExLLaMA-v2 / GPTQ-for-LLaMa):
- Tile X into shared memory; load once and reuse across multiple N tiles
- CUDA graph to eliminate launch overhead
- Mixed-precision accumulation + warp shuffle reduction

## LinearW4A16 Interface

```python
# Quantize from an existing Linear layer
layer = LinearW4A16.from_float(linear_module, group_size=128)

# forward
out = layer(x)  # x: [*, K] bf16 → out: [*, N] bf16
```

`from_float` runs offline quantization via `_per_group_minmax_quantize`; `forward` calls `w4a16_gemm`.
