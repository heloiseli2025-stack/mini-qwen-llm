# M3 W4A16 AWQ 量化

## 目标

将所有 Linear 层权重从 bf16 压缩到 int4，通过减少 weight HBM 读取量（4x 压缩）加速 decode。

---

## 数据格式与 packing 约定

### int4 打包规则（nibble little-endian）

```
packed[i] = w[8i+0] | w[8i+1]<<4 | w[8i+2]<<8 | ... | w[8i+7]<<28
```

nibble 0 = 最低 4 bit = 最低 index。pack/unpack 必须用同一套规则，否则数值会乱。

### Tensor 布局

| Tensor | Shape | Dtype | 备注 |
|--------|-------|-------|------|
| `qweight` | `[K//8, N]` | int32 | 沿 K 维打包；`qweight[j,n]` 的 nibble i = weight[8j+i, n] |
| `scales`  | `[G, N]`    | bf16  | 每组 scale，G = K // group_size |
| `qzeros`  | `[G, N//8]` | int32 | 沿 N 维打包；同 pack_int4 约定 |

---

## Dequant 公式

```
w_fp = (w_int.to(fp32) - zero.to(fp32)) * scale.to(fp32)
```

asymmetric quantization，zero 是 int4 值（0~15）。全程 fp32 避免 bf16 减法精度损失。

---

## Triton Kernel 设计（w4a16_gemm）

### Grid

```
Grid = (cdiv(M, BLOCK_M), cdiv(N, BLOCK_N))
```

每个 program 负责 `out[pid_m*BM:(pid_m+1)*BM, pid_n*BN:(pid_n+1)*BN]`。

### 内层循环结构

```python
for g in range(K // group_size):          # 外层：group 循环（K//128 次）
    load scale [BLOCK_N]
    load + decode qzero [BLOCK_N]
    for j in static_range(group_size // 8):  # 内层：constexpr 展开 16 次
        load qw [BLOCK_N] int32
        unpack 8 nibbles → w_fp0..w_fp7 [BLOCK_N] fp32
        load x0..x7 [BLOCK_M] bf16→fp32
        acc += xi[:, None] * w_fpi[None, :]  # outer product 累加
```

### 关键设计决策

1. **K=8 outer product 而非 tl.dot**：SM 8.9 Tensor Core 要求 K≥16，K=8 时 dot 会 fallback 到 CUDA Core。改用 8 次 outer product 语义等价，且 decode 是 memory-bound，不依赖 Tensor Core。

2. **Shift 用 literal 常量**：`(qw >> 0) & 0xF`, `(qw >> 4) & 0xF`, ...，保证 Triton 编译时展开（shift 为 constexpr）。

3. **dequant 全程 fp32**：先转 fp32 再减再乘，避免 bf16 直接减法引入的精度损失（scale 为 bf16，直接减 zero 会损失 1~2 bit）。

4. **Autotune key 含 group_size**：`key = ['M', 'N', 'K', 'group_size']`，不同 group size 可能最优 tile 不同。

## AWQ 量化理论（简化版，用于测试）

本实现使用 per-group min-max quantization（不含 activation-aware scaling）作为 kernel 验证数据源。

```python
scale = (w_max - w_min) / 15.0          # [G, N]
zero  = (-w_min / scale).round()         # [G, N] int4，0~15
w_int = round(w / scale + zero)          # [K, N] int4
```

完整 AWQ（Lin et al., 2023）在此基础上加入 activation-aware weight scaling：
1. 用 calibration 数据计算每列 x 的激活幅值 `s_x`
2. 对 w 每列乘以 `s_x^α`（α ≈ 0.5）做 pre-scaling
3. 对 x 补偿除以 `s_x^α`（可 fold 进前一层 RMS Norm）
4. 目的：把对精度影响大的列（激活幅值大）的量化误差压低

此实现骨架接口预留（`awq.py` 中 `_per_group_minmax_quantize`），完整 calibration 流程作为扩展。

## 实测数字（RTX 4090，K=1024，N=4096）

### 正确性（pytest）

| 测试用例 | max abs error | 结论 |
|---------|---------------|------|
| M=1, K=128, N=8（最小 shape） | 0.000000 | PASS |
| M=1, K=1024, N=4096 | ≤ 0.004 | PASS |
| M=4, K=1024, N=4096 | ≤ 0.250 | PASS |
| M=16, K=1024, N=4096 | ≤ 0.500 | PASS |

误差来源：fp32 kernel 与 fp64 reference 在 bf16 rounding boundary 上的 1-ULP 差异，magnitude ~130 时 1-ULP = 1.0，K=1024 累加后最大差 ~0.5 ULP。真正的 dequant bug（nibble 顺序错误）会给 O(4+) 误差，与此截然不同。

### 性能（decode 场景）

| M | BF16 cuBLAS | W4A16 Triton | speedup |
|---|-------------|--------------|---------|
| 1 | 0.008 ms | 0.051 ms | 0.16x |
| 4 | 0.009 ms | 0.058 ms | 0.15x |
| 8 | 0.009 ms | 0.057 ms | 0.16x |
| 16 | 0.009 ms | 0.058 ms | 0.16x |

**当前实现未达到 ≥ 2x 性能目标。**

瓶颈分析：
- BF16 cuBLAS 读 8 MB weight 约 8 μs，接近 HBM 带宽极限（4090 ~1 TB/s）
- Triton kernel 理论应读 2 MB（int4 → 4x 压缩 → 2 μs），实测 51 μs（M=1）
- cuBLAS 极度优化（CUTLASS 后端、persistent kernel、L2 prefetch），Triton 无法匹敌
- 差距主要来自：Triton launch overhead（每次 launch ~5 μs），以及 cuBLAS 深度 cache 复用

已实施的优化（M2→M3 迭代）：
1. **GEMM kernel**：X 改为 `[BLOCK_M, 16]` 2D coalesced load + `tl.dot(K=16)`，取代 8 次独立 scalar load
2. **GEMV kernel（M=1 专用）**：独立 1D kernel 避免 BLOCK_M=16 浪费，M=1 latency 从 175 μs → 51 μs（3.4x 提升）
3. **K=16 组合**：将两行 qweight（各 8 nibble）用 `tl.where` 合并，满足 Triton `tl.dot` K≥16 要求

**production 级别进一步优化方向**（ExLLaMA-v2 / GPTQ-for-LLaMa 的做法）：
- X tile 进 shared memory，一次加载供多个 N tile 复用
- CUDA graph 消除 launch overhead
- 混合精度 accumulation + warp shuffle reduction

## LinearW4A16 接口

```python
# 从已有 Linear 量化
layer = LinearW4A16.from_float(linear_module, group_size=128)

# forward
out = layer(x)  # x: [*, K] bf16 → out: [*, N] bf16
```

`from_float` 用 `_per_group_minmax_quantize` 做离线量化，forward 调用 `w4a16_gemm`。
